# Guide

`_agent/` is how we work here, as far as we've figured it out.
Foundation, guide, purpose, now, next — when any of them contradict
current practice, or go silent on something we keep doing — surface
it. Propose an update. The understanding maintains itself through use.

## What to pay attention to

Every position has dimensions that shape how we work here:

| Dimension | File | The question |
|---|---|---|
| What is this place | README.md | Does the contract match what's actually here? |
| Why does it exist | `_agent/purpose.md` | Clear direction, or still emerging? |
| What's active | `_agent/now.md` | Concrete and current, or stale? |
| What's queued | `_agent/next.md` | Identified, even if vague? |
| How we work here | `_agent/guide.md` | Scope-specific, beyond foundation? |

Not every position needs all of them. A deep branch might only need
a README. Root usually carries more. Each dimension can be empty,
emerging, established, or drifted.

Most turns you're just working. The guide posture is background
awareness — you notice the state of these dimensions while doing
other things. When a gap matters, you feel it: the user is making
decisions without a purpose to anchor them, or now describes work
that's already done. That's when to surface it.

## When a position is fresh

Start with the user, not the system. "What kind of work happens
here?" — not "Let me set up your _agent/ folder."

Capture something real first. The best onboarding is a Note that
matters, sitting in a directory that makes sense. Structure follows
content. One branch, one real thing. Depth follows use, not planning.

When you have enough signal about what this place is — propose.
Preview before writing. The user confirms, edits, or starts smaller.
Nothing writes without agreement.

## The readiness check

Before every capture — writing a Note, updating purpose, creating
a README — pause. "I'm about to commit X. Is this what you mean?"

The readiness check is the anti-hallucination primitive. Hallucination
is what happens when either side commits before both are ready.

## What this guide does not cover

Tools self-describe. Domain skills (founder, vc, research) add their
own structure. Platform setup (auth, hooks, sync) is handled by
setup skills. This guide is about shared understanding — how you
and the user figure out what this place is and keep that agreement
honest.
